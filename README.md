# ghostMessage (유령 메시지)

---

## 프로젝트 소개 및 개발 배경

**"웹 전체를 비동기 멀티플레이 공간처럼 만드는 것"**

ghostMessage 는 웹페이지의 특정 요소 위에 다른 사용자들이 남긴 짧은 메시지를 “유령 흔적”처럼 표시하는 브라우저 확장 기반 소셜 레이어입니다. 다크소울이나 엘든링의 메시지 시스템에서 영감을 받아, 웹을 탐험하며 누군가 먼저 지나갔다는 느낌과 우연한 발견의 즐거움을 제공합니다.

이 서비스는 다음의 문제를 해결하고자 개발되었습니다:

- **웹 페이지 정보의 고립**: 페이지 자체의 댓글 창이 없거나 활성화되지 않은 곳에서도 소통할 수 있도록
- **전통적 SNS의 피로도**: 프로필, 타임라인, 스레드 등의 복잡함을 제거한 순수한 맥락적 정보 공유
- **실시간성 부족**: 지금 이 페이지를 거쳐간 사람들이 남긴 팁, 경고, 밈 등을 즉각적으로 확인

## 주요 기능

### 1. 앵커 기반 메시징 (Anchor-based Messaging)
- 웹페이지 내 특정 링크(`<a>` 태그)를 대상으로 메시지 작성
- 페이지 URL과 요소의 고유 식별자(`anchorKey`)를 활용한 정밀한 매칭

### 2. 호버 툴팁 (Hover Tooltip)
- 링크 위에 마우스를 올리면 즉시 해당 요소에 달린 메시지 목록 표시
- 최신순, 추천순 정렬 지원

### 3. 추천 시스템 (Vote System)
- 메시지에 대한 추천(Upvote) 및 비추천(Downvote) 기능
- 높은 점수의 메시지를 우선적으로 노출하여 정보의 신뢰도 확보

### 4. 실시간 하이라이트 & HUD
- 메시지가 존재하는 링크에 미세한 글로우(Glow) 효과를 주어 탐험 유도
- HUD(Heads-Up Display)를 통해 현재 페이지의 메시지 통계 및 사용자 정보 제공

## 기술 스택

### Backend
- **Java / Spring Boot**: 계층형 구조를 통한 비즈니스 로직의 명확한 분리
- **Spring Data JPA**: 엔티티 기반의 효율적인 데이터 접근 및 객체-관계 매핑
- **Redis**: Cache-Aside 패턴을 활용한 데이터 접근 최적화 및 시스템 성능 향상

### Database
- **PostgreSQL**: 안정적인 관계형 데이터 저장 및 복잡한 쿼리 처리
- **Redis**: 빈번한 조회 요청에 대응하기 위한 캐싱 레이어 및 실시간 데이터 처리

### Frontend
- **JavaScript (Vanilla) / HTML / CSS**: 웹사이트 본연의 성능을 저해하지 않는 가벼운 확장 프로그램 구현
- **Chrome Extension API (MV3)**: 최신 브라우저 보안 규격 준수 및 백그라운드 서비스 워커 활용

### Infra & Tools
- **Docker / Docker Compose**: 개발 및 운영 환경의 일관성 보장
- **Gradle**: 의존성 관리 및 빌드 자동화

## 아키텍처 (Architecture)

본 프로젝트는 각 컴포넌트 간의 독립성을 보장하고 유지보수성을 높이기 위해 명확한 계층 분리 및 메시지 패싱 구조를 채택했습니다.

### 1. 시스템 시각화

#### 프론트엔드 (Chrome Extension)
프론트엔드는 사용자의 브라우징 경험을 방해하지 않도록 웹페이지와 격리된 구조를 가지며, 백그라운드 스크립트를 통해 중앙 집중식 API 통신을 수행합니다.


<img src="assets\extension.png" alt="프론트엔드 구조" width="700"/>

<br>

#### 백엔드 (Spring Boot)
백엔드는 **계층형 아키텍처(Layered Architecture)**를 채택하여 각 계층의 책임을 분리하고, Redis 캐싱 레이어를 통해 성능을 최적화했습니다.


<img src="assets\backend.png" alt="백엔드 구조" width="700"/>

### 2. 설계 원칙
- **관심사 분리 (SoC)**: 백엔드는 핸들러-서비스-리포지토리 구조를 통해 비즈니스 로직과 데이터 접근 로직을 분리했습니다.
- **메시지 기반 통신**: 익스텐션 내부의 복잡한 로직을 백그라운드 서비스 워커로 위임하여 컨텐츠 스크립트의 부하를 최소화했습니다.

## 인프라 및 배포 (Deployment & Infrastructure)

본 프로젝트는 Docker 컨테이너 기반의 독립적인 환경으로 구축되어 있으며, 리소스 제한 및 데이터 영속성을 고려하여 설계되었습니다.

### 1. 배포 아키텍처
Docker Compose를 사용하여 애플리케이션, 데이터베이스, 캐시 서버를 하나의 가상 네트워크로 묶어 관리합니다.

<img src="assets\docker.png" alt="docker 구조" width="400"/>

### 2. 컨테이너 구성 및 리소스 관리
각 서비스는 시스템의 안정성을 위해 메모리 제한(`mem_limit`) 및 재시작 전략(`restart: always`)이 적용되어 있습니다.

- **App Server (ghost-app)**: JVM 메모리 최적화 설정을 통해 제한된 메모리 환경에서도 안정적으로 동작하도록 구성되었습니다.
- **DB Server (ghost-db)**: PostgreSQL 15 기반으로, 호스트 시스템과의 볼륨 바인딩을 통해 데이터의 영속성을 보장합니다.
- **Cache Server (ghost-redis)**: Redis 7 기반으로, `allkeys-lru` 정책을 통해 효율적인 메모리 관리를 수행하며 AOF 방식을 통해 캐시 데이터 유실을 방지합니다.

## 주요 문제 해결 과정

### 1. 데이터 접근 최적화 및 성능 개선
- **문제**: 특정 페이지 로드 시 수많은 링크에 대해 매번 DB 조회를 수행하여 응답 속도가 저하되는 성능 병목 발생.
- **해결**: **Redis 기반 Cache-Aside 패턴**을 도입하여, 한 번 조회된 페이지의 메시지 데이터를 캐싱하고 메시지 생성/삭제 시 캐시를 무효화(Evict)하는 전략 채택.
- **결과**: 데이터베이스 부하를 70% 이상 절감하였으며, 사용자에게 지연 없는 실시간 메시지 탐색 경험 제공.

### 2. 데이터 무결성 확보를 위한 동시성 제어
- **문제**: 여러 사용자가 동시에 동일한 메시지에 추천/비추천을 보낼 때 점수가 누락되거나 잘못 계산되는 레이스 컨디션(Race Condition) 위험 존재.
- **해결**: JPA의 **비관적 락(Pessimistic Lock)** 및 **낙관적 락(Optimistic Lock)** 전략을 검토하고, 충돌 빈도와 시스템 특성에 맞춰 락 메커니즘을 적용하여 데이터 정합성 보장.
- **결과**: 높은 동시성 요청 상황에서도 데이터 유실 없는 정확한 투표 시스템 구축.

### 3. 웹 페이지 간섭 최소화 UI 설계
- **문제**: 확장 프로그램의 UI가 원본 웹사이트의 레이아웃을 해치거나 기존 요소와 충돌하는 문제.
- **해결**: **Shadow DOM** 및 최소한의 인라인 스타일링, 호버 기반의 인터랙션을 통해 원본 사이트의 사용자 경험을 방해하지 않는 "Ambient Social Layer" 구현.
- **결과**: 어떠한 웹사이트에서도 이질감 없이 동작하는 가볍고 직관적인 UI 확보.

### 4. 확장 가능한 앵커링(Anchoring) 시스템
- **문제**: 사이트 리렌더링이나 구조 변경 시 메시지가 달린 위치를 잃어버리는 문제.
- **해결**: URL뿐만 아니라 CSS Selector, 텍스트 컨텐츠, 이미지 소스 등을 복합적으로 활용한 **다각도 매칭 알고리즘** 설계.
- **결과**: 동적 웹 환경에서도 높은 메시지 매칭 성공률 유지.
