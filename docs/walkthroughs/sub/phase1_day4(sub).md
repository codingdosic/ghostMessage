# Phase 1 - Day 4(sub): k6 부하 테스트와 저사양 환경 성능 최적화

이 문서는 **1GB RAM x86_64** 서버에서 **k6**와 **Redis**를 활용해 시스템의 한계를 테스트하고, 열악한 자원 환경을 기술적으로 극복하는 과정을 다룹니다.

---

## 1. 전략적 선택: 저사양 서버에서의 성능 분석

### 면접 대비: "1GB RAM 서버에서 부하 테스트가 유의미한가요?"
> **답변 핵심**: "오히려 사양이 낮기 때문에 **성능 최적화의 가시성**이 더 명확하게 나타납니다. 넉넉한 서버에서는 드러나지 않던 DB 커넥션 병목이나 메모리 부족 현상이 1GB 환경에서는 즉각적으로 발생합니다. 저는 **k6**를 로컬에서 실행하여 서버에 부하를 주는 방식으로 서버 자원을 온전히 서비스 처리에 집중시켰고, **Redis** 도입 전후의 응답 속도와 에러율 변화를 수치로 증명했습니다."

---

## 2. 실무 워크스루 (Step-by-Step)

### Step 1: 원격(Remote) 부하 테스트 환경 구축
서버의 리소스를 아끼기 위해 k6 테스트는 로컬 PC에서 서버로 부하를 가하는 방식으로 진행합니다.

1.  **로컬 PC에 k6 설치**: [k6 공식 가이드](https://k6.io/docs/getting-started/installation/) 참고.
2.  **테스트 시나리오 작성**: `docs/phase1_day4.md`의 시나리오를 사용하되, 대상 주소를 오라클 서버의 **공인 IP**로 설정합니다.
3.  **Baseline 측정**: Redis가 적용되지 않은 순수 DB 조회 상태에서 `p95 Latency`와 `Success Rate`를 기록합니다.

### Step 2: Redis 최적화 및 1GB RAM 생존 전략
메모리가 부족하므로 Redis 설정을 최소한으로 유지합니다.

1.  **캐시 만료(TTL) 설정**: 메모리 점유를 최소화하기 위해 모든 Redis 키에 짧은 TTL(예: 1시간)을 부여합니다.
2.  **Eviction Policy 설정**: `maxmemory-policy allkeys-lru`를 설정하여 메모리가 꽉 차면 가장 오래된 데이터를 자동으로 삭제하도록 합니다.
3.  **결과 확인**: Redis 적용 후 DB 쿼리 부하가 감소하여, 1GB RAM임에도 불구하고 더 많은 동시 접속자(VUs)를 처리할 수 있게 된 수치를 확보합니다.

## 3. 기술 면접 예상 질문

**Q: 부하 테스트를 서버 내부가 아닌 로컬에서 실행한 이유는 무엇인가요?**
**A**: "1GB RAM이라는 제한적인 자원 환경 때문입니다. 부하를 생성하는 k6 프로세스 자체가 상당한 CPU와 메모리를 소모하므로, 테스트 대상 서버의 자원을 온전히 서비스 성능 측정에 사용하기 위해 외부(로컬 PC)에서 부하를 발생시키는 방식을 선택했습니다."

**Q: 저사양 서버에서 Redis를 도입할 때 가장 우려되는 점은 무엇이었나요?**
**A**: "Redis 자체가 사용하는 메모리 점유율입니다. 이를 해결하기 위해 **Redis의 maxmemory 제한**을 128MB로 타이트하게 설정하고, **LRU(Least Recently Used) 알고리즘**을 적용하여 메모리 포화 시에도 시스템이 다운되지 않고 효율적으로 캐시를 교체하도록 설계했습니다."
