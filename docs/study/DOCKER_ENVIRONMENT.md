# Web Ghost Messages 프로젝트 도커 운영 환경 상세 분석

본 문서는 프로젝트의 도커(Docker) 기반 운영 환경을 기술적 관점에서 상세히 분석한 자료입니다.

---

## 1. 도커(Docker)의 기초 이해
도커는 애플리케이션을 **'표준화된 컨테이너'**에 담아 어디서든 동일하게 실행하게 해주는 가상화 플랫폼입니다.

- **핵심 개념**:
    - **이미지(Image)**: 설계도(Blueprint). 애플리케이션을 실행하는 데 필요한 모든 파일과 설정의 스냅샷입니다.
    - **컨테이너(Container)**: 실행 중인 인스턴스. 설계도를 바탕으로 실제로 동작하는 개별 프로세스입니다. 컨테이너는 서로 독립적인(Isolated) 환경을 가집니다.
    - **Docker Compose**: 여러 개의 컨테이너를 한 번에 관리(생성, 삭제, 실행)하기 위한 도구입니다.

---

## 2. 프로젝트 아키텍처 상세 분석
현재 프로젝트는 5개의 컨테이너가 서로 긴밀하게 네트워크로 연결되어 동작합니다.

| 서비스명 | 이미지명 | 역할 | 주요 동작 원리 |
| :--- | :--- | :--- | :--- |
| **app** | ghostmessage_server | API 서버 (Spring Boot) | 요청 처리, 비즈니스 로직 수행 |
| **db** | postgres:15-alpine | 데이터 저장소 (PostgreSQL) | 영속적인 데이터 보관 |
| **redis** | redis:7-alpine | 캐시 서버 | 메시지 응답 속도 향상 |
| **pgadmin** | dpage/pgadmin4 | DB 관리 UI | 브라우저를 통한 DB 관리 |
| **tunnel** | cloudflare/cloudflared | 보안 터널링 | 외부 인터넷과 서버 연결 |

### 데이터 흐름
1. 사용자가 익스텐션에서 요청 전송.
2. Cloudflare 터널을 통해 서버의 `app` 컨테이너로 데이터 전달.
3. `app` 컨테이너는 필요에 따라 `db`(데이터) 또는 `redis`(캐시)와 통신하여 결과값 생성.

---

## 3. 서버 하드웨어 및 자원 파악

### 물리적 환경 (호스트 머신)
- **메모리(RAM)**: 약 956MiB (시스템 총 메모리)
- **디스크(Storage)**: 45GB (루트 파티션)

### 리소스 할당 및 최적화 상태
- **메모리 제한(Memory Limit)**: 
    - `app` 컨테이너는 스왑 현상을 방지하기 위해 `JAVA_OPTS`로 JVM 힙 메모리를 `128MiB(Xms) ~ 256MiB(Xmx)`로 조정하였습니다. 이는 물리 RAM이 부족한 환경(총 1GB 이하)에서 서버 안정성을 확보하기 위한 핵심 설정입니다.
- **디스크 상태**: 현재 17GB를 사용 중(36%)으로, 향후 확장성을 고려해도 충분한 여유 공간을 확보하고 있습니다.

---

## 4. 도커 운영의 핵심 원리

### 1) 프로세스 격리 (Isolation)
각 컨테이너는 자신만의 파일 시스템, 네트워크, 메모리 공간을 가집니다. `ghost-app`이 오류로 죽더라도 `ghost-db`는 안전하게 동작합니다.

### 2) 데이터 영속성 (Persistence)
도커 컨테이너는 삭제되면 내부 데이터도 함께 사라집니다. 이를 방지하기 위해 **볼륨(Volume)**을 사용합니다.
- `ghostMessage_server/data/postgres`: DB 데이터 저장
- `ghostMessage_server/data/redis`: Redis 데이터 저장
- 이 폴더들은 컨테이너 외부(호스트 머신)에 저장되므로, 컨테이너를 재시작해도 데이터는 안전하게 보존됩니다.

### 3) 통신 (Networking)
컨테이너들은 서로 가상의 내부 네트워크로 연결됩니다.
- `app` 컨테이너는 외부 네트워크 포트(8080) 외에도 내부 네트워크를 통해 `db` 컨테이너의 5432 포트로 직접 접근합니다. (`application.yml`의 `jdbc:postgresql://db:5432/ghost_db` 참조)

## 5. 상세 설정 파일 분석: docker-compose.yml

`docker-compose.yml`은 프로젝트의 모든 컨테이너를 한꺼번에 실행하고 관리하는 '설계도'입니다. 주요 설정과 그 의도는 다음과 같습니다.

### 5.1 서비스 정의 (`services`)

#### `app` (Spring Boot API Server)
- **`build: .`**: 현재 폴더의 `Dockerfile`을 사용하여 애플리케이션 이미지를 직접 빌드합니다.
- **`environment`**: DB/Redis 접속 정보를 하드코딩하지 않고 환경 변수로 주입합니다. 이는 **보안(비밀 정보 노출 방지)**과 **환경 간 유연한 설정 변경**을 위함입니다.
- **`depends_on`**: `db`와 `redis`가 먼저 실행되어야 `app`이 시작되도록 순서를 제어합니다.

#### `db` (PostgreSQL)
- **`volumes: - ./data/postgres:/var/lib/postgresql/data`**: 컨테이너 내부의 데이터를 호스트 머신의 `./data/postgres` 폴더에 저장합니다. 컨테이너를 삭제해도 **데이터가 영구 보존**되는 핵심 설정입니다.
- **`image: postgres:15-alpine`**: `alpine` 버전은 용량이 극도로 작아 서버 자원 소모를 최소화합니다.

#### `redis` (Cache)
- **`command: ... --maxmemory 64mb --maxmemory-policy allkeys-lru`**: Redis가 사용하는 메모리를 최대 64MB로 제한하고, 꽉 찼을 경우 가장 오래된 데이터부터 자동으로 삭제하도록 설정하여 **서버 메모리 과부하를 예방**합니다.

---

### 5.2 공통 설정의 의도

- **`restart: always`**: 서버가 예상치 못하게 재시작되거나 컨테이너가 죽었을 때, 도커 데몬이 **자동으로 컨테이너를 재시작**하여 서비스 가용성을 보장합니다.
- **`mem_limit`**: 각 컨테이너별로 사용할 수 있는 메모리 상한을 명시합니다. 한 컨테이너가 폭주해도 서버 전체가 멈추는 것을 방지하기 위한 **안전 장치**입니다.
