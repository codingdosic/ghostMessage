# Portfolio Strategy: 기술 고도화 및 아키텍처 전략

본 문서는 현재 프로젝트를 채용 담당자나 기술 평가자에게 어필할 수 있는 수준의 **"기술적 고민과 해결 사례"**로 전환하기 위한 로드맵입니다.

## 1. 포트폴리오 관점의 기본 원칙 (The "Portfolio Way")
모든 개선 작업은 아래의 3단계를 거쳐 기록되어야 합니다.
1.  **Problem Definition (문제 정의)**: 어떤 상황에서 어떤 기술적 병목(Latency, Load, Consistency)이 발생했는가? (예: "투표 요청 시 DB 락 경합으로 인한 응답 시간 지연")
2.  **Architectural Justification (기술적 근거)**: 왜 하필 그 기술(예: Redis)인가? 비용과 복잡도를 고려했을 때 왜 타당한가? (예: "RDB의 쓰기 부하를 줄이기 위해 캐싱 계층 도입, Cache-Aside 패턴 채택")
3.  **Measurable Impact (수치적 성과)**: 개선 전/후의 성능 지표(TPS, Latency, 서버 리소스 사용량)를 숫자로 증명. (예: "k6 부하 테스트 결과, TPS 300% 상승")

---

## 2. 3대 기술 고도화 상세 로드맵

### 항목 1: 데이터 접근 최적화 (Performance & Caching)
*   **시나리오**: 특정 페이지 조회 시 다수의 메시지를 매번 DB에서 조회하는 비효율 발생.
*   **구현 계획**: 
    *   **Redis** 도입 (Cache-Aside 패턴).
    *   `MessageService`에서 DB 조회 전 Redis를 먼저 확인하도록 로직 수정.
    *   메시지 생성/삭제/투표 시 `RedisTemplate`을 사용하여 캐시 무효화(`evict`) 처리.
*   **포트폴리오 포인트**: Cache-Aside 패턴의 동기화 전략(쓰기 시 무효화) 및 성능 개선 지표.

### 항목 2: 동시성 제어 및 시스템 안정성 (Reliability & Testing)
*   **시나리오**: 여러 유저가 동일 메시지에 투표할 때 `upVoteScore`가 유실되는 레이스 컨디션 발생 가능성 + 한 유저가 동시에 여러개의 탭으로 동시에 작성 시 하나의 카운트만 올라가는 문제.
*   **구현 계획**: 
    *   **k6** 부하 테스트 스크립트 작성.
    *   JPA **Pessimistic Lock** , **Optimistic Lock** 적용 비교.
    *   테스트 통과 후 성공 사례 정리.
*   **포트폴리오 포인트**: 부하 테스트를 통한 문제 재현 과정과 락 전략 적용을 통한 데이터 무결성 확보.

### 항목 3: 서비스 신뢰성 및 무결성 강화 (Security & Rate Limiting)
*   **시나리오**: 특정 IP 또는 사용자 ID가 단시간에 대량의 요청(DoS, 스팸 작성)을 보내 서버 자원을 점유.
*   **구현 계획**: 
    *   Spring Boot **Bucket4j** 라이브러리 도입.
    *   사용자별/IP별 호출 한도(Rate Limit) 설정.
*   **포트폴리오 포인트**: 서비스 가용성 확보 전략 및 어뷰징 차단 구현.
