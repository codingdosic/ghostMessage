# Redis 캐싱 적용 및 성능 최적화 계획 (Redis Implementation Plan)

본 문서는 1GB RAM의 제한된 서버 환경에서 발생하는 성능 병목(Tail Latency)을 해결하기 위해 Redis 캐싱을  도입하는 과정과 그 결과를 기록합니다.

---

## 1. 도입 배경 및 근거 (Rationales)

### 1.1. 부하 테스트 결과 분석 (Pre-Redis Benchmark)
VUs(가상 사용자) 증가에 따른 성능 변화를 측정한 결과, 명확한 자원 병목 현상이 관찰되었습니다.

| 지표 | 50 VUs (30s) | 100 VUs (30s) | 변화율 |
|:---:|:---:|:---:|:---:|
| **평균 응답 시간 (avg)** | 520.87 ms | 1.18 s | **126.5% 증가** |
| **중간 응답 시간 (med)** | 419.52 ms | 1.08 s | **157.4% 증가** |
| **상위 95% 지연 (p95)** | 1.29 s | 3.1 s | **140.3% 증가** |
| **최대 지연 (max)** | 2.38 s | 7.11 s | **198.7% 증가** |
| **초당 요청 수 (reqs/s)** | 57.18 reqs/s | 63.68 reqs/s | 11.3% 증가 |

### 1.2. 병목 현상 진단 (Bottleneck Analysis)
1.  **자원 한계 도달:** VUs가 2배 증가했음에도 초당 처리량(reqs/s)은 11% 증가에 그쳤으며, 응답 시간은 2 배 이상 폭증했습니다. 이는 서버 자원(CPU/Memory/DB Connection)이 이미 포화 상태에 도달했음을 의미합니다.
2.  **Tail Latency 심화:** 100 VUs 상황에서 최대 지연 시간이 7초를 상회합니다. 이는 저사양 서버(1GB RAM)에서 DB I/O 부하가 몰릴 때 JVM 가비지 컬렉션(GC)이 빈번하게 발생하거나, DB 커넥션 풀 대기 시간이 길어지 면서 발생하는 전형적인 병목 현상입니다.
3.  **캐싱의 필연성:** 현재 API는 단순 조회 위주임에도 부하에 매우 취약합니다. DB 접근을 최소화하는 Redis 캐싱을 통해 응답 시간을 낮추고 처리량을 확보해야 합니다.

## 2. 최종 캐싱 대상 API (Final Target APIs)

| 순위 | API 엔드포인트 | 중요도 | 도입 근거 (Rationale) |
|:---:|:---|:---:|:---|
| **1** | `GET /api/messages/all` | **최상** | HUD 업데이트용. 100 VUs 시 p95 3초 이상 지연 발생의 주범. |
| **2** | `GET /api/messages` | **상** | UX 핵심. 툴팁 반응성 확보(평균 1.18s -> 0.1s 이하 목표). |     
| **3** | `GET /api/users/{uuid}` | **중** | 매 페이지 로드 시 중복 조회 발생. DB 커넥션 점유율 감소 유 도. |

## 3. 구현 단계 (Workflow)

### Step 1: 인프라 및 의존성 설정
*   `build.gradle`에 `spring-boot-starter-data-redis` 추가.
*   `application.yml`에 Redis 접속 정보 및 CacheManager 설정.

### Step 2: 캐싱 로직 적용
*   `MessageService` 및 `UserService` 조회 메서드에 `@Cacheable` 적용.
*   `pageUrl`, `anchorKey`, `uuid`를 캐시 키로 활용.

### Step 3: 데이터 정합성 보장
*   메시지 저장/삭제/투표 시 관련 캐시 무효화(`@CacheEvict`) 로직 구현.

### Step 4: 성능 개선 검증
*   Redis 적용 후 50/100 VUs 재테스트 수행 및 지표 비교.

## 4. 성능 측정 및 비교 (k6 Benchmark)

### 4.1. Redis 적용 전 (Pre-Redis Summary)
*   **상태:** 가용성 100%이나 부하 증가 시 응답 시간 폭증.
*   **특이사항:** 100 VUs에서 p95 3.1s, Max 7.1s로 실서비스 이용 시 불편함 초래 가능 수준.

### 4.2. Redis 적용 후 (Post-Redis Summary)

VUs 부하 상황에서 Redis 캐싱 도입으로 인한 성능 개선 지표를 확인했습니다.

| 지표 | Redis 미적용 (100 VUs) | Redis 적용 (100 VUs) | 개선율 |
|:---:|:---:|:---:|:---:|
| **평균 응답 시간 (avg)** | 1.18 s | **437.49 ms** | **약 63% 속도 향상** |
| **상위 95% 지연 (p95)** | 3.10 s | **981.91 ms** | **약 68% 지연 감소** |
| **최대 지연 (max)** | 7.11 s | **2.18 s** | **약 69% 렉 완화** |
| **초당 처리량 (reqs/s)** | 63.68 reqs/s | **126.40 reqs/s** | **약 2배 처리량 증가** |

### 5. 최종 결론 및 기술적 인사이트 (Insight)

1.  **Tail Latency의 성공적 제어:**
    *   100 VUs 상황에서 상위 5%의 사용자가 겪던 3초 이상의 답답한 지연 시간을 1초 미만(981ms)으로 안정 화시켰습니다.
    *   이는 단순히 평균 속도를 높인 것을 넘어, 시스템의 예측 가능성과 사용자 경험(UX)의 일관성을 확보했음을 의미합니다.

2.  **물리적 자원 한계의 소프트웨어적 극복:**
    *   1GB RAM이라는 극히 제한된 서버 환경에서 PostgreSQL과 JVM 간의 메모리 경합 문제를 Redis라는 "메모리 방패"를 통해 해결했습니다.
    *   DB I/O를 메모리 캐싱으로 대체함으로써 서버 증설 없이 처리 능력을 2배(Throughput Double-up) 향상 시키는 성과를 거두었습니다.

3.  **데이터 정합성과 성능의 트레이드오프 최적화:**
    *   `Cache Aside` 전략과 `@CacheEvict`를 통한 즉시 무효화 로직을 결합하여, 성능 향상과 동시에 데이터의 정확성을 완벽하게 유지했습니다.

## 6. 데이터 무결성 보장을 위한 동시성 제어 (Concurrency Control)

고성능 캐싱 도입과 더불어, 데이터의 정확성을 보장하기 위해 서비스의 특성에 따라 차별화된 락(Lock) 전략을 적용했습니다.

### 6.1. 투표 기능: 비관적 락 (Pessimistic Write Lock)
- **적용 대상**: `MessageRepository.findByIdWithLock(id)`
- **적용 근거**: 추천/비추천 투표는 High-Contention이 빈번한 작업으로, 조회 후 수정 과정에서 레이스 컨디션 발생 시 점수 유실(Lost Update) 위험이 큽니다.
- **구현 방식**: JPA의 `@Lock(LockModeType.PESSIMISTIC_WRITE)`를 사용하여 데이터 조회 시점에 DB 레벨에서 행(Row) 잠금을 수행하여 트랜잭션 간 정합성을 완벽하게 보장합니다.

### 6.2. 사용자 작성 제한: 낙관적 락 (Optimistic Lock)
- **적용 대상**: `User` 엔티티 (일일 메시지 작성 횟수)
- **적용 근거**: 사용자가 메시지 작성 횟수를 초과하지 않도록 제한하는 로직은 경합 확률이 상대적으로 낮습니다. 비관적 락을 사용할 경우 불필요한 성능 저하가 발생할 수 있어 성능 효율을 중시했습니다.
- **구현 방식**: `User` 엔티티에 `@Version` 필드를 추가하여, 수정 시 커밋 시점에 버전 불일치 확인을 통해 동시 수정을 방지하고 예외를 발생시켜 데이터 무결성을 유지합니다.
