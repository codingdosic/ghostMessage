# 동시성 제어 및 데이터 무결성 확보 전략 (Concurrency Control Strategy)

본 문서는 다수의 사용자가 동시에 투표(추천)하거나 메시지를 작성하는 상황에서 데이터의 정합성을 보장하기 위한 락(Lock) 전략과 그 기술적 근거를 정리합니다.

---

## 1. 락 메커니즘 비교: 비관적 락 vs 낙관적 락

| 구분 | 비관적 락 (Pessimistic Lock) | 낙관적 락 (Optimistic Lock) |
|:---:|:---|:---|
| **개념** | 데이터 충돌이 발생할 것으로 가정하고 우선적으로 잠금. | 데이터 충돌이 적을 것으로 가정하고 커밋 시점에 확인. |
| **구현 방식** | DB 레벨의 `SELECT ... FOR UPDATE` 구문 사용. | 엔티티 내 `@Version` 필드를 통한 애플리케이션 레벨 체크. |
| **장점** | 데이터 정합성이 완벽히 보장됨. 충돌 시 대기 후 순차 처리. | DB 자원 점유가 없어 성능상 유리하며 데드락 위험이 낮음. |
| **단점** | 동시 처리량 저하(Blocking), 데드락 발생 가능성 존재. | 충돌 시 예외가 발생하며, 별도의 재시도 로직 구현 필요. |

---

## 2. 투표(추천) 기능에 락이 필요한 이유 (Lost Update 문제)

투표 기능은 **'조회(Read) → 수정(Modify) → 저장(Write)'** 단계를 거치며, 락이 없을 경우 **갱신 유실(Lost Update)** 현상이 발생합니다.

### 2.1. 문제 상황 시나리오
1. **사용자 A와 B**가 동시에 현재 추천 수가 **10**인 메시지를 조회합니다.
2. **사용자 A**가 추천 수에 +1을 더해 **11**로 계산합니다. (아직 DB 반영 전)
3. **사용자 B** 또한 기존 값 10에 +1을 더해 **11**로 계산합니다.
4. 두 사용자의 수정 요청이 차례로 반영되면, 결과는 **12가 아닌 11**이 되어 한 명의 투표가 유실됩니다.

### 2.2. 해결 근거
- 인기 있는 웹페이지의 링크는 수천 명의 사용자가 동시에 투표할 수 있는 **High-Contention** 지점입니다.
- 데이터의 유실은 서비스의 신뢰도와 직결되므로, 조회 시점부터 행(Row)을 선점하여 동시 수정을 차단하는 **비관적 락** 적용이 타당합니다.

---

## 3. 서비스 도메인별 락 적용 가이드라인

### 3.1. 락 적용이 권장되는 케이스
*   **메시지 추천/비추천 (Vote Score)**:
    - **이유**: 동일 행에 대한 빈번한 수정 발생.
    - **전략**: **비관적 락 (Pessimistic Lock)**. 데이터 정합성을 최우선으로 하여 점수 유실 방지.
*   **사용자별 일일 활동 제한 (Daily Limits)**:
    - **이유**: 한 명의 사용자가 여러 창에서 동시에 요청을 보내 한도(예: 하루 10개 작성)를 초과할 가능성 방지.
    - **전략**: **낙관적 락 (Optimistic Lock)**. 동일 사용자의 중복 요청 확률은 상대적으로 낮으므로 성능 효율성 중시.

### 3.2. 락이 불필요하거나 과도한 케이스
*   **메시지 작성 (Insert)**: 새로운 행이 삽입되는 과정이므로 기존 데이터와의 충돌이 없습니다.
*   **메시지 조회 (Read)**: 다수의 사용자가 동시에 읽더라도 데이터 변경을 일으키지 않으므로 락이 필요 없습니다.
*   **메시지 삭제 (Delete)**: 삭제 권한 체크 후 단발성으로 수행되므로 경합 발생 가능성이 매우 낮습니다.

---

## 4. 기술적 인사이트 및 결론

1.  **정합성 vs 성능의 트레이드오프**: 모든 곳에 락을 거는 것은 성능을 저해합니다. 수정 빈도와 비즈니스 중요도에 따라 락의 종류를 차등 적용하는 것이 중요합니다.
2.  **저사양 서버 환경의 특수성**: 1GB RAM 환경에서는 DB 커넥션 유지 시간이 길어지면 시스템 전체가 마비될 수 있습니다. 따라서 락의 범위를 최소화하고, 캐싱 레이어(Redis)와 적절히 조합하여 부하를 분산해야 합니다.
