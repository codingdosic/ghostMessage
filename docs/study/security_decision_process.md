# 기술 의사결정 보고서: 영구 세션 기반 익명 서비스의 보안 전략

## 1. 배경 및 현황
Ghost Message 프로젝트는 웹 브라우저 익스텐션을 통해 사용자 식별자(UUID)를 발급하고, 이를 기반으로 별도의 로그인 절차 없이 메시지를 읽고 쓰는 '영구 세션' 구조를 가지고 있다.

- **현재 식별 방식**: 익스텐션 설치 시 서버에서 발급받은 UUID를 `chrome.storage.local`에 저장하여 사용.
- **보안 취약점**: UUID가 유일한 식별자이자 인증 수단이므로, 타인의 UUID 탈취 시 해당 사용자로 위장하여 메시지 작성/삭제 등 모든 권한 행사가 가능함.
- **사용자 경험(UX) 요구사항**: 익명 서비스의 특성상 가입/로그인 등의 번거로운 절차를 지양하고 '가벼운 사용성'을 유지해야 함.

## 2. 보안 대안 비교 분석

### 대안 A: JWT (JSON Web Token) 도입
- **방식**: 로그인 시 서버에서 짧은 주기의 Access Token과 긴 주기의 Refresh Token을 발급하여 매 요청마다 검증.
- **장점**: 업계 표준이며 보안성이 높고 토큰 탈취 시 피해 범위가 시간적으로 제한됨.
- **단점**: 
    - 만료 시 재인증(로그인)이 필요하여 '영구 세션'의 편리함과 충돌함.
    - 익명 서비스에서 '계정 생성'이라는 초기 진입 장벽이 발생함.
    - 토큰 갱신 로직 등 클라이언트/서버 구현 복잡도 증가.

### 대안 B: 보안 코드 (Security Code) 도입 (최종 선택)
- **방식**: UUID와 함께 생성된 6자리 고유 코드를 '복구 및 인증 키'로 활용.
- **장점**:
    - 별도의 로그인 UI 없이 기존의 '무설정(Zero-config)' 사용성 유지 가능.
    - 기기 변경이나 재설치 시에도 UUID와 보안 코드 조합으로 계정 복구 가능.
    - 민감한 API 요청 시 보안 코드를 자동 포함하여 UUID 탈취 공격 방어 가능.
- **단점**: 보안 코드가 유출될 경우 UUID 방식과 동일한 위험이 있으나, 사용자에게 '보안 키'로서의 인식을 심어줄 수 있음.

## 3. 의사결정 근거
본 프로젝트는 **"보안의 엄격함"**보다는 **"사용의 편리함과 익명성"**에 더 높은 가치를 둔다. 따라서 다음과 같은 이유로 **보안 코드 방식**을 채택한다.

1. **UX 최우선**: 사용자가 "로그인"이라는 행위를 인지하지 못하게 하면서도 보안을 강화할 수 있는 유일한 방법임.
2. **영구 세션 최적화**: 만료 개념이 없는 영구 세션 구조에서 JWT의 만료/갱신 메커니즘은 불필요한 비용임.
3. **단계적 강화 가능**: 초기에는 복구용으로 도입하고, 이후 민감 API 검증용으로 확장하기 용이함.

## 4. 향후 로드맵 (단계적 보안 강화)

### Phase 1: 기반 마련 및 계정 복구 (현재)
- 유저 엔티티에 `security_code` 필드 추가.
- 설정 페이지에 내 정보(UUID, 보안 코드) 확인 및 계정 복구 UI 구현.
- 사용자는 재설치 후에도 이전의 UUID와 보안 코드를 입력하여 자신의 데이터를 되찾을 수 있음.

### Phase 2: 민감 동작 인증 강화 (차기)
- 메시지 작성, 삭제, 투표 등 '쓰기' 권한이 필요한 API 호출 시 UUID와 보안 코드를 함께 전송.
- 서버는 두 값이 DB 상의 정보와 일치할 때만 요청을 승인.
- 클라이언트는 저장된 보안 코드를 자동으로 첨부하므로 사용자 인터랙션의 변화는 없음.
