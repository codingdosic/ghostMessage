# Project GhostMessage 분석 및 업데이트 리포트 (2026-05-25)

본 문서는 GhostMessage 프로젝트의 코드베이스 분석, 서버 인프라 현황, 그리고 향후 기능 업데이트 시 사용자 계정에 미치는 영향에 대한 분석 내용을 정리한 리포트입니다.

---

## 1. 프로젝트 개요
GhostMessage는 웹상의 특정 링크에 사용자들의 "유령 메시지"를 남기고 공유할 수 있는 시스템입니다. 크롬 익스텐션을 통해 UI를 제공하고, 스프링 부트 기반의 백엔드에서 데이터를 관리합니다.

## 2. 기술 스택 및 아키텍처

### 2.1 Frontend (Chrome Extension)
- **Manifest V3:** 최신 크롬 익스텐션 표준 준수.
- **Core Engine:** `MutationObserver`를 활용한 동적 링크 감지 및 하이라이팅.
- **UI Components:** Shadow DOM을 활용하여 웹페이지 스타일 간섭 없이 모달 및 툴팁 렌더링.
- **Communication:** Content Script <-> Background Service Worker <-> Backend API 구조.

### 2.2 Backend (Spring Boot)
- **Framework:** Spring Boot 3.x
- **Database:** PostgreSQL 15 (영구 데이터 저장)
- **Cache:** Redis 7 (페이지별/툴팁별 메시지 목록 캐싱)
- **Concurrency:** 투표 시스템의 정합성을 위해 JPA 비관적 락(`PESSIMISTIC_WRITE`) 적용.
- **Rate Limiting:** 사용자별 일일 메시지 작성(10회) 및 투표(20회) 제한 로직 포함.

---

## 3. 서버 인프라 상태 (Docker)
현재 서버는 Ubuntu 환경에서 Docker Compose를 통해 3개의 컨테이너로 운영 중입니다.

| 서비스 명 | 이미지 | 포트 | 메모리 제한 | 주요 역할 |
| :--- | :--- | :--- | :--- | :--- |
| `ghost-app` | `app-app` | 8080 | 512MB | API 서버 및 비즈니스 로직 처리 |
| `ghost-db` | `postgres:15-alpine` | 5432 | 256MB | 메시지, 유저, 투표 데이터 저장 |
| `ghost-redis` | `redis:7-alpine` | 6379 | 128MB | 조회 성능 향상을 위한 캐싱 |

## 4. 사용자 계정 영향 분석 (UUID 기반)

### 4.1 현재 구조
- **식별 방식:** 별도의 로그인 없이 익스텐션 최초 설치 시 서버에서 발급받은 **UUID**를 `chrome.storage.local`에 저장하여 사용자를 식별합니다.
- **장점:** 사용자 편의성이 높고 익명성이 보장됩니다.

### 4.2 업데이트 시 계정 유실 위험 요소
기능 업데이트나 시스템 변경 시 의도치 않게 UUID가 재할당되어 이전 계정 정보를 유실할 수 있는 위험 시나리오는 다음과 같습니다.

1.  **익스텐션 스토리지 키 변경:** `chrome.storage.local`에 저장된 `userId` 키 이름을 변경할 경우, 기존 값을 찾지 못해 새 계정을 생성하게 됩니다.
2.  **익스텐션 삭제 후 재설치:** 브라우저 정책상 익스텐션 삭제 시 로컬 저장소 데이터가 함께 삭제됩니다.
3.  **브라우저 데이터 초기화:** 사용자가 브라우저 캐시 및 사이트 데이터를 모두 삭제할 경우 UUID가 유실될 수 있습니다.
4.  **백엔드 마이그레이션 오류:** DB 스키마 변경 중 유저 테이블 데이터가 유실되거나 UUID 형식이 일치하지 않게 될 경우 매칭이 불가능해집니다.

### 4.3 대응 가이드
- **스토리지 키 고정:** `userId` 키 이름을 절대적으로 유지하며, 변경 필요 시 반드시 마이그레이션 로직을 선행해야 합니다.
- **데이터 백업 고려:** 향후 유실 방지를 위해 익명 계정을 소셜 계정 등으로 연동할 수 있는 브릿지 기능을 고려할 수 있습니다.
- **유효성 체크:** 익스텐션 구동 시 현재 UUID가 서버에 존재하는지 확인하는 핸드셰이크 과정을 통해 상태를 조기에 파악해야 합니다.

---

## 5. 결론
현재 프로젝트는 안정적인 Docker 기반 인프라와 효율적인 캐싱 구조를 갖추고 있습니다. 향후 업데이트 시 **익스텐션의 로컬 스토리지 키 관리**와 **백엔드의 데이터 정합성**에 주의한다면, 별도의 로그인 시스템 없이도 사용자 계정의 연속성을 충분히 유지할 수 있습니다.
