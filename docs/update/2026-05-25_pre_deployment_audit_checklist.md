# 배포 전 필수 개선 사항 및 보안 체크리스트

본 문서는 GhostMessage 프로젝트를 실제 운영 환경에 배포하기 전, 안정성과 보안, 그리고 관리 편의성을 위해 반드시 해결해야 할 핵심 과제들을 정리한 체크리스트입니다.

---

## 1. 데이터베이스 관리 편의성 개선 (DB 관리 UI 확보)

현재 터미널(`docker exec`)을 통해서만 DB를 관리하는 방식은 운영 효율이 매우 낮습니다.

- **문제점:** 데이터 오기입 수정, 불량 메시지 수동 삭제, 유저 활동 모니터링 시 매번 복잡한 SQL 쿼리를 터미널에 입력해야 함.
- **해결 방안:** `pgAdmin4` 또는 `Adminer`를 Docker Compose에 추가하여 웹 기반 GUI 환경 구축.
- **예상 작업:**
  - `docker-compose.yml`에 pgAdmin 서비스 추가.
  - 외부 포트(예: 5050)를 통해 브라우저에서 DB 접속 설정.

## 2. 보안 취약점 보완 (Minimum Security)

소규모 프로젝트라도 자동화된 봇 공격이나 악의적인 요청으로부터 최소한의 방어가 필요합니다.

- **CORS 정책 제한:**
  - **현재:** 모든 도메인(`*`) 허용.
  - **변경:** 실제 익스텐션 ID(`chrome-extension://[ID]`)만 허용하도록 `WebConfig` 또는 Controller 수정.
- **환경 변수 관리:**
  - **현재:** `docker-compose.yml`에 DB 비밀번호가 평문(`password`)으로 노출됨.
  - **변경:** `.env` 파일을 사용하거나 배포 시 환경 변수를 통해 복잡한 비밀번호로 교체.
- **인증 로직의 부재:**
  - **현황:** UUID만 알면 타인의 메시지를 삭제하거나 투표를 조작할 수 있는 구조.
  - **대응:** 당장 고도화된 인증(JWT 등)이 어렵다면, 중요 API(삭제 등) 호출 시 최소한의 요청 출처 검증이라도 강화해야 함.

## 3. 운영 환경의 기술적 요구사항 (HTTPS)

크롬 익스텐션은 보안상의 이유로 `https://` 통신을 강력히 권장하며, 배포 시 필수적인 요소가 될 수 있습니다.

- **SSL 인증서 적용:** `Certbot`을 이용해 API 서버에 HTTPS 적용.
- **API 주소 전환:** 익스텐션 내 `background.js`의 `SERVER_URL`을 HTTPS 주소로 변경.

---

## [우선순위 순 작업 목록]

1.  **[High]** `docker-compose.yml` 보안 설정 (비밀번호 변경 및 pgAdmin 추가).
2.  **[High]** 백엔드 CORS 설정 제한 (`*` 제거).
3.  **[Low]** 서버 도메인 연결 및 HTTPS(SSL) 적용.

---

## 결론
위 항목들은 단순히 기능을 구현하는 것을 넘어, **"실제로 서비스가 돌아가게 만드는"** 운영의 필수 요소들입니다. 특히 DB 관리 UI 확보와 CORS 제한은 배포 직후 발생할 수 있는 관리적 혼란과 보안 위협을 막는 가장 효율적인 조치가 될 것입니다.
